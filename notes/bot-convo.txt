  ---
  iiac-perf Design (0.2.0)

  Goal

  Measure latency distributions of communication primitives in Rust вҖ” not just averages, but full histograms (p50, p90, p99, p99.9, etc.) to reveal tail
  latency behavior.

  Measurement Approach

  Ping-pong (round-trip): Sender sends a payload, receives it back. Simple, no clock synchronization needed, captures what most users care about. One-way
  measurement can come later.

  The payload problem: The user correctly identified that trivial work could stay in registers. Strategy:
  - Payload wraps a Vec<u8>, configurable size (default 64 bytes = one cache line)
  - Use std::hint::black_box on both input and output to prevent compiler elision
  - "Work" is identity/echo вҖ” just return the payload
  - For techniques needing owned data (channels), clone is included in the measurement since real usage would require it too

  Timing:
  - std::time::Instant for portability (~25ns resolution on Linux)
  - For very fast operations (<100ns per call, like bare function calls), the harness auto-batches: times N calls together, reports per-batch histogram
  - Calibration phase detects when batching is needed
  - Individual timing for everything else (channels, IPC, network)

  Core Trait Design

  Two related traits вҖ” sync and async вҖ” sharing a common base:

  trait Technique: Send {
      fn name(&self) -> &str;
      fn description(&self) -> &str;
      fn kind(&self) -> TechniqueKind; // Sync or Async
      fn setup(&mut self) -> Result<()>;
      fn teardown(&mut self) -> Result<()>;
  }

  trait SyncTechnique: Technique {
      fn round_trip(&mut self, payload: &Payload) -> Result<Payload>;
  }

  trait AsyncTechnique: Technique {
      fn round_trip(&mut self, payload: &Payload)
          -> impl Future<Output = Result<Payload>> + Send;
  }

  Why two traits instead of one? Forcing async techniques through block_on misses the point вҖ” the interesting cost of async is the .await within a running
  runtime. And wrapping sync techniques in async adds noise. The harness has separate sync/async run loops but shares all histogram and reporting code.

  Harness Design

  For each technique:
    1. setup()
    2. Warmup phase (default 10,000 iterations, not recorded)
    3. Calibration: run 1000 iterations, measure per-iteration time
       - If < 100ns: switch to batch mode (batch_size = 100ns / per_iter_time)
    4. Measurement phase (default 1,000,000 iterations)
       - Record each iteration (or batch) duration into HdrHistogram
    5. teardown()
    6. Report results

  Histogram via hdrhistogram crate (Rust port of Gil Tene's HdrHistogram). Configured for 3 significant digits, range 1nsвҖ“1s.

  CLI

  iiac-perf [OPTIONS] [TECHNIQUES...]

  Options:
    -n, --iterations <N>     Number of iterations [default: 1000000]
    -w, --warmup <N>         Warmup iterations [default: 10000]
    -p, --payload-size <N>   Payload size in bytes [default: 64]
    -o, --output <FORMAT>    Output format: text, json [default: text]
    --all                    Run all available techniques

  Techniques:
    fn_call                  Direct function call
    async_fn_call            Async function call (.await in tokio runtime)
